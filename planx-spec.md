## PipelineSpec v4 Contract

**PipelineSpec v4** is the ONLY canonical declarative format for Planx pipelines.

### Structure

PipelineSpec follows Kubernetes-style resource conventions:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `apiVersion` | string | YES | Must be `planx.io/v4` |
| `kind` | string | YES | Must be `Pipeline` |
| `metadata.name` | string | YES | Unique pipeline identifier |
| `metadata.tenantId` | string | YES | Owning tenant identifier |
| `spec.source` | SourceSpec | YES | Exactly one source node |
| `spec.processors` | []ProcessorSpec | NO | Ordered list of processors |
| `spec.sink` | SinkSpec | YES | Exactly one sink node |

Each node (source/processor/sink) contains:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | YES | Unique node identifier within pipeline |
| `plugin` | string | YES | Plugin name (must exist in registry) |
| `config` | object | NO | Plugin-specific configuration (opaque) |

### Serialization

PipelineSpec is **serialization-agnostic**:
- YAML is the preferred human-authoring format
- JSON is equivalent and fully supported
- No behavioral difference exists between formats

### Authorship

PipelineSpec is **human-authored**:
- Users create and edit specifications directly
- Specifications are declarative intent, not execution instructions
- Tooling (UI, CLI, API) MAY generate specifications on behalf of users

## Specification Lifecycle

```
Authoring Spec → Control Plane API → Runtime Plan → Executor
```

| Stage | Artifact | Owner |
|-------|----------|-------|
| Authoring | PipelineSpec | User / Tooling |
| API Submission | CreatePipelineRequest | Control Plane |
| Resolution | RuntimePipelinePlan | Resolver |
| Execution | Session state | Executor |

### Critical Rules

1. **Runtime plans are derived artifacts**
   - RuntimePipelinePlan is generated by the Resolver
   - Users MUST NOT author runtime models directly
   - Runtime plans are immutable once generated

2. **Control Plane does not modify specifications**
   - CreatePipelineRequest embeds PipelineSpec without transformation
   - tenantId/projectId are Control Plane metadata, not spec fields

## Config Opaqueness Contract

The `config` field in each node specification is an **opaque JSON object**.

| Rule | Description |
|------|-------------|
| Engine MUST NOT parse | Config contents are never interpreted by planx-engine |
| Engine MUST serialize | Config is marshaled to `[]byte` for runtime delivery |
| Plugin MUST validate | Schema validation is the plugin's responsibility |
| Plugin MUST document | Config schema lives in plugin documentation |

The Engine treats config as:
```go
Config map[string]any `yaml:"config,omitempty"`
```

The Runtime treats config as:
```go
Config []byte // Opaque configuration
```

## Non-Goals / Explicitly Out of Scope

The following are **NOT** part of the PipelineSpec v4 user-facing contract:

| Item | Reason |
|------|--------|
| Runtime execution models | Derived artifact, not user-authored |
| Binary paths | Deployment concern, resolved at runtime |
| gRPC addresses / endpoints | Resolved by plugin manager |
| Session IDs | Generated at execution time |
| Executor topology (DAG scheduling) | Internal engine implementation |
| Metrics backend configuration | Infrastructure concern |
| Retry policies | Not implemented; future scope |
| Circuit breaker configuration | Not implemented; future scope |

Users MUST NOT attempt to specify these in PipelineSpec.
