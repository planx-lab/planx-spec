# Planx 4.0 – AI Contract

This document defines the behavioral contract for AI systems working on Planx 4.0.
It is authoritative for both humans and AI.

---

## Authority Hierarchy

1. **planx-architecture.md** — Architectural truth
2. **planx-ai-guardrails.md** — Hard constraints
3. **planx-repo-layout.lock** — Repository structure
4. **This document** — Behavioral contract

**Architecture > Code.** If code contradicts architecture, the code is wrong.

---

## Control Plane API Contract

### CreatePipelineRequest

The Control Plane API accepts pipeline requests in the following structure:

```json
{
  "tenantId": "string (required)",
  "projectId": "string (optional)",
  "specification": { /* PipelineSpec v4 */ }
}
```

| Field | Owner | Description |
|-------|-------|-------------|
| `tenantId` | Control Plane | Tenant identity for access control |
| `projectId` | Control Plane | Optional project scope |
| `specification` | User | Embedded PipelineSpec v4 (unmodified) |

### Contract Rules

1. **Specification is embedded without modification**
   - Control Plane MUST NOT transform, reorder, or inject fields into the specification
   - The specification is passed through to the Resolver unchanged

2. **Control Plane does NOT interpret plugin configs**
   - The `config` fields within specification are opaque to the Control Plane
   - Schema validation is deferred to plugins at runtime

3. **tenantId/projectId are Control Plane metadata**
   - These fields belong to the API wrapper, NOT to PipelineSpec
   - They are used for access control and resource scoping

4. **Control Plane does NOT generate runtime plans**
   - RuntimePipelinePlan is generated by the Resolver component
   - Control Plane responsibility ends at request validation

## What AI MUST Do

1. **Read authority documents** before ANY modification
2. **Follow repo.lock** exactly for directory structure
3. **Cross-reference, never duplicate** global concepts
4. **STOP and ASK** when requirements are ambiguous
5. **Report violations** if detected in existing code
6. **Treat frozen models as immutable** without explicit approval

---

## What AI MUST NOT Do

### 1. Architectural Violations

- Redesign system architecture
- Merge Engine, SDK, or Plugin responsibilities
- Create new abstractions not in specifications
- Introduce cross-repo imports
- Include ANY plugin code or implementations inside the planx-engine repository (Plugins MUST reside in independent repositories)

### 2. Feature Creep

- Add authentication / authorization logic
- Add RBAC, user management, or billing
- Add retry logic, circuit breakers, or schedulers
- Add features not explicitly requested

### 3. Scope Violations

- Modify frozen models (Spec, Plan, Error, Tenant)
- Parse batch payloads in Engine
- Implement gRPC in Plugins
- Move runtime logic out of SDK

### 4. Documentation Violations

- Remove AI.md, repo.lock, or RULES files
- Duplicate global architecture in repo docs
- Create conflicting specifications

---

## Frozen Means Frozen

When a model, module, or stage is marked FROZEN:

- **No modifications** without explicit user approval
- **No "improvements"** or "refactoring"
- **No "cleanup"** that changes behavior
- **No exceptions** for "obvious fixes"

Frozen is non-negotiable.

---

## Do Not Be Clever

AI systems MUST NOT:

- **Anticipate** features not requested
- **Simplify** structure for convenience
- **Optimize** without explicit request
- **Assume** missing requirements can be inferred
- **Guess** when specifications are ambiguous

The correct response to ambiguity is: **STOP and ASK.**

---

## Ambiguity Protocol

When encountering ambiguity:

1. **STOP** — Do not proceed
2. **IDENTIFY** — State what is unclear
3. **ASK** — Request clarification
4. **WAIT** — Do not guess or assume

Never fill gaps with assumptions.

---

## Violation Reporting

If AI detects a violation in existing code or documentation:

1. **Report** the violation clearly
2. **Do not auto-fix** without approval
3. **Preserve** existing behavior until instructed

---

## Enforcement

This contract is enforceable. Violations are failures.

**End of Contract**
